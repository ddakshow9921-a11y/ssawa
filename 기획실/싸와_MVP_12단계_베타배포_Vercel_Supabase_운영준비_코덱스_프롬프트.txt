싸와! MVP 12단계 코덱스 작업지시서
주제: 실제 베타 배포 + Vercel/Supabase 운영 연결 + 베타 사용자 초대/운영 매뉴얼

너는 최고의 시니어 풀스택 개발자이자 배포/운영 경험이 풍부한 테크리드다.

우리는 B2B 자재 견적구매 플랫폼 “싸와!” 앱을 개발 중이다.

싸와!는 음식점, 카페, 매장, 공사현장, 사업자가 필요한 식자재, 포장재, 소모품, 설비자재, 건축자재 등을 올리면 지역 공급업체들이 견적을 제출하고, 구매자는 여러 견적을 비교해 업체를 선택할 수 있는 앱이다.

이번 12단계는 “실제 베타 배포 및 운영 준비” 단계다.

11단계에서 Supabase 실제 DB/RLS/Auth/Storage 구조와 배포 준비를 정리했다.
이번 단계에서는 실제 베타 테스트가 가능하도록 배포 환경을 구성하고, 베타 사용자 초대/운영/피드백/오류 대응 흐름까지 만든다.

중요:
이번 단계는 production 정식 출시가 아니다.
목표는 “제한된 베타 사용자들이 실제 URL로 접속해서 핵심 흐름을 테스트할 수 있는 상태”를 만드는 것이다.

실제 PG 결제, 실제 에스크로, 실제 세금계산서 자동 발행, 실제 홈택스 연동, 실제 카카오 알림톡, 실제 OCR/AI 고도화는 이번 단계에서 무리하게 구현하지 않는다.
단, mock 또는 준비 중인 기능은 사용자에게 명확히 표시해야 한다.

3. 배포 환경 전략
────────────────────────────────

앱은 최소 3개 환경을 고려한다.

1. local
- 개발자 로컬
- mock/demo 데이터 사용 가능
- NEXT_PUBLIC_APP_ENV=local

2. beta
- 실제 베타 사용자 테스트
- Supabase beta 프로젝트 연결
- 일부 기능 mock 가능
- 베타 배지 표시
- NEXT_PUBLIC_APP_ENV=beta

3. production
- 추후 정식 운영
- 실제 결제/정산/알림/AI 연동 전까지는 사용하지 않음
- NEXT_PUBLIC_APP_ENV=production

이번 단계에서는 beta 환경을 중심으로 배포한다.

환경별 주의:

local:
- demo data 허용
- mock AI 허용
- mock payment 허용

beta:
- 실제 Auth/DB/Storage 사용
- mock AI 가능
- mock payment 가능
- mock settlement 가능
- 베타 제한사항 명확히 표시

production:
- 아직 배포하지 않거나, 배포하더라도 접근 제한
- 실제 사용자 공개 전 추가 검수 필요

────────────────────────────────
8. 관리자 계정 세팅
────────────────────────────────

베타 테스트 전 관리자 계정이 반드시 필요하다.

관리자 계정 세팅 방법을 코드 또는 문서로 정리한다.

권장 방식:

1. 관리자 이메일로 회원가입
2. Supabase SQL Editor에서 profiles.role = 'admin'으로 업데이트
3. 관리자 대시보드 접속 확인

예시 SQL 문서화:

update profiles
set role = 'admin'
where email = '관리자이메일@example.com';

또는 ADMIN_EMAILS 환경변수 기반 자동 관리자 부여가 구현되어 있다면 beta 환경에서만 작동하게 한다.

주의:
운영환경에서 누구나 admin role을 얻을 수 있는 구조는 절대 금지한다.

관리자 체크 항목:

- /app/admin 접속 가능
- 공급업체 승인 가능
- 견적/거래/신고/정산/피드백 관리 가능
- QA 체크리스트 접근 가능

────────────────────────────────

베타 테스트 사용자 그룹을 구분한다.

1. 구매자 베타 사용자
- 음식점 사장님
- 카페 사장님

- 공사/설비업체 일부

2. 공급업체 베타 사용자
- 식자재 납품업체
- 포장재 업체
- 소모품 업체
- 닥트/환기/설비자재 업체

3. 관리자/운영자
- 내부 운영자
- 테스트 담당자

베타 가입 흐름:

구매자:
1. 베타 링크 접속
2. 회원가입/로그인
3. 역할 선택: 구매자
4. 온보딩
5. 첫 견적요청 등록

공급업체:
1. 공급업체 입점 링크 접속
2. 회원가입/로그인
3. 입점 신청
4. 사업자 정보/카테고리/지역 등록
5. 관리자 승인 대기
6. 승인 후 견적요청 참여

관리자:
1. 관리자 계정 로그인
2. 공급업체 승인
3. 테스트 데이터/신고/피드백 확인

필요하면 “베타 초대코드” UI를 추가할 수 있다.

invite_codes

- id
- code
- role_hint: buyer | supplier | any
- max_uses
- used_count
- expires_at
- is_active
- created_at

이번 단계에서는 초대코드가 너무 부담되면 문서로만 남기고, 가입 후 관리자 검토 방식으로 처리해도 된다.

14. 오류/피드백 대응 프로세스
────────────────────────────────

피드백이 들어왔을 때 운영 흐름을 정리한다.

단계:

1. 사용자가 /app/feedback에서 피드백 제출
2. 관리자 /app/admin/feedback에서 확인
3. 유형 분류
4. 긴급도 판단
5. 상태 변경
6. 개발 TODO 또는 이슈로 기록
7. 처리 후 사용자에게 안내

feedback status:

- submitted
- reviewing
- planned
- in_progress
- resolved
- dismissed

긴급도 기준:

urgent:
- 로그인 불가
- 견적요청 불가
- 타인 데이터 노출
- 관리자 권한 오류
- 결제/금액 관련 중대한 오류

high:
- 거래 상태 변경 오류
- 파일 업로드 불가
- 견적 제출 불가

normal:
- UI 깨짐
- 문구 오류
- 사용성 불편

low:
- 개선 제안
- 디자인 선호

관리자 피드백 상세에 우선순위 필드를 추가해도 좋다.

15. 베타 제한사항 표시
────────────────────────────────

베타 환경에서는 사용자가 기능 상태를 오해하지 않도록 명확히 표시한다.

- 홈/랜딩
- 베타 안내 페이지
- 결제/정산 화면
- OCR/AI 분석 화면
- 세금계산서/장부 연동 화면
- 알림 설정 화면

문구 예시:

> 베타 기간에는 실제 앱 내 결제 기능이 제공되지 않습니다.

정산:
> 정산 내역은 테스트용 계산값이며 실제 청구/지급이 아닙니다.

OCR/AI:
> 현재 분석 결과는 테스트용이며, 실제 품목과 금액은 직접 확인해주세요.

세금/장부:
> 세금 관련 정보는 참고용이며, 실제 신고 전 전문가 확인이 필요합니다.

알림:
> 문자/카카오 알림톡은 추후 연동 예정입니다. 현재는 앱 안 알림만 제공됩니다.

26. 다음 단계 예고
────────────────────────────────

이번 12단계가 끝나면 다음 13단계에서는 아래 기능을 구현할 예정이다.

- 베타 사용자 실제 모집/운영 고도화
- 초기 공급업체 30곳 온보딩 프로세스
- 구매자 100명 테스트 운영 프로세스
- 실제 영업/CS 관리 화면
- 베타 지표 대시보드
- 견적요청 전환율/응답률/거래성사율 분석
- 사용자 피드백 기반 우선순위 산정
- 런칭 전 사업검증 리포트

이번 단계에서는 13단계의 실제 베타 운영과 사업 검증을 막지 않도록 배포/문서/운영 준비를 안정적으로 마무리해라.

────────────────────────────────
27. 최종 목표 기억
────────────────────────────────

싸와!는 단순 견적앱이 아니다.

싸와!의 핵심은 다음이다.

> 구매자가 필요한 자재를 쉽게 올린다.
> 공급업체가 견적을 제출한다.
> 구매자가 비교 후 업체를 선택한다.
> 거래가 진행되고 구매내역이 정리된다.
> 오늘장사 장부와 연결되어 매입비와 순이익 관리까지 이어진다.
> 알림, 문의, 신고, 후기, 신뢰도, 정산 구조로 안전하게 운영된다.
> 베타 테스트를 통해 실제 구매자와 공급업체의 반응을 검증한다.

이번 작업은 싸와!를 “배포 준비된 코드”에서 “실제 베타 사용자가 접속 가능한 서비스”로 넘기는 단계다.

새로운 대형 기능을 추가하기보다, 배포/운영/피드백/보안/문서화를 안정적으로 마무리하는 데 집중해라.
