싸와! MVP 6단계 코덱스 작업지시서
주제: OCR/AI 품목 자동분석 + 거래명세서/견적서 업로드 자동입력 고도화

너는 최고의 시니어 풀스택 개발자이자 UX 기획자다.

우리는 B2B 자재 견적구매 플랫폼 “싸와!” 앱을 개발 중이다.

싸와!는 음식점, 카페, 매장, 공사현장, 사업자가 필요한 식자재, 포장재, 소모품, 설비자재, 건축자재 등을 올리면 지역 공급업체들이 견적을 제출하고, 구매자는 여러 견적을 비교해 업체를 선택할 수 있는 앱이다.

이전 단계까지 다음 기능을 구현했다.

1단계:
- 싸와! 홈 화면
- 구매자 견적요청 작성
- 견적요청 목록
- 견적요청 상세
- 공급업체 견적 제출
- 구매자 견적 비교
- 관리자 기본 관리 화면

2단계:
- 견적요청 단계형 UX
- 사진/거래명세서/파일 업로드 UI
- 문장 입력 → 품목 리스트 변환 mock 기능
- AI 품목 정리 mock 화면
- 견적요청 템플릿
- 지난 요청 다시 사용하기
- 요청 완성도 점수
- 공급업체 매칭 예상 안내 UI

3단계:
- 공급업체 입점 신청
- 공급업체 사업자 정보 등록
- 공급업체 취급 카테고리 설정
- 공급업체 납품 가능 지역 설정
- 관리자 공급업체 승인/반려
- 공급업체 프로필
- 지역/카테고리 기반 요청 필터링
- 공급업체 신뢰 정보 표시

4단계:
- 구매자 견적 선택
- 거래 생성
- 거래 상세
- 거래 상태 관리
- 구매자 거래내역
- 공급업체 거래관리
- 관리자 거래 모니터링
- 구매내역 저장 구조
- 오늘장사 장부 연동 준비

5단계:
- 거래 완료 후 구매내역 자동 생성
- 오늘장사 장부 반영 준비
- 매입비 자동 분류
- 매입비/절감액 리포트
- 세금계산서 상태 관리
- 거래명세서/영수증 자료함
- 월별 매입 리포트
- 오늘장사 연동 상태 관리

이번 작업은 MVP 6단계다.

이번 단계의 핵심 목표는 “사용자가 직접 품목을 입력하지 않아도 되는 구조”를 만드는 것이다.

즉, 사장님이 기존 거래명세서, 견적서, 영수증, 품목 사진, 카톡 주문 내용, 엑셀 파일 등을 올리면 앱이 OCR/AI 분석 흐름을 통해 품목명, 규격, 수량, 단위, 금액, 공급업체명, 세금계산서 여부 등을 자동으로 추출하고, 사용자는 검토 후 견적요청서 또는 구매내역으로 변환할 수 있어야 한다.

이번 단계에서는 실제 AI/OCR API 연동을 해도 되고, 프로젝트 환경상 API 키나 백엔드가 준비되지 않았다면 mock 분석 엔진과 API 연결 가능한 구조를 만들어라.

중요한 것은 실제 서비스에 연결 가능한 구조와 UX 흐름을 안정적으로 만드는 것이다.

카톡 주문내역:
- 복사한 주문 내용을 품목 리스트로 정리합니다.

품목 사진:
- 사진 속 물건을 기준으로 견적요청을 만들 수 있습니다.

9. mock 분석 엔진
────────────────────────────────

실제 OCR/AI API가 준비되지 않았다면 mock 분석 엔진을 구현한다.

권장 함수:

- createAnalysisJob(input)
- runMockAnalysis(job)
- parseTextToAnalysisItems(text)
- extractAmountsFromText(text)
- detectCategoryFromItems(items)
- calculateAnalysisConfidence(items, metadata)
- convertAnalysisToQuoteRequest(job, items)
- convertAnalysisToPurchaseRecord(job, items)

텍스트 파싱 예시:

입력:
닭정육 20kg 120,000원
양파 15kg 28,000원
식용유 18L 2통 76,000원
배달봉투 대형 1000장 45,000원

출력:
- 닭정육 / 20 / kg / 120000원
- 양파 / 15 / kg / 28000원
- 식용유 / 18L / 2 / 통 / 76000원
- 배달봉투 / 대형 / 1000 / 장 / 45000원

완벽할 필요는 없다.
중요한 것은 분석 결과를 사용자가 검토하고 수정할 수 있는 구조다.

12. 분석 결과를 견적요청으로 변환
────────────────────────────────

사용자가 분석 결과를 견적요청으로 변환할 수 있어야 한다.

버튼:

> 이 품목으로 견적요청 만들기

변환 흐름:

1. 사용자가 분석 품목을 확인한다.
2. 견적요청으로 보낼 품목을 선택한다.
3. 카테고리, 배송지역, 희망납품일, 세금계산서 여부 등을 입력한다.
4. quote_request와 quote_request_items를 생성한다.
5. 기존 견적서/거래명세서 파일은 quote_attachments에 연결한다.
6. 견적요청 상세로 이동한다.

변환 화면 문구:

> 기존 거래내역을 기준으로 더 저렴한 견적을 받아보세요.

표시 옵션:

- 모든 품목 포함
- 선택한 품목만 포함
- 금액 정보 숨기고 견적요청
- 기존 금액을 비교 기준으로 저장

중요:
공급업체에게 기존 거래처의 민감한 가격을 보여줄지 여부는 사용자가 선택해야 한다.

옵션:
- 기존 금액을 공급업체에게 공개하지 않음
- 기존 금액을 참고가로 공개
- 기존 총액만 공개
- 기존 품목/수량만 공개

기본값은 “기존 금액 비공개”로 한다.

────────────────────────────────
13. 분석 결과를 구매내역으로 변환
────────────────────────────────

분석 결과를 오늘장사 장부 반영 준비용 purchase_record로 변환할 수 있어야 한다.

버튼:

> 구매내역으로 저장하기

변환 흐름:

1. 거래처/품목/금액을 확인한다.
2. 매입 카테고리를 선택한다.
3. 세금계산서 상태를 선택한다.
4. 결제수단을 선택한다.
5. purchase_records를 생성한다.
6. 오늘장사 장부 반영 대기 상태로 저장한다.

purchase_record 기본값:

- accounting_status: pending
- tax_invoice_status: requested | issued | none | pending
- source: analysis

화면 문구:

> 이 자료를 구매내역으로 저장하면 오늘장사 장부에 반영할 수 있습니다.

19. 오늘장사 장부 연동 준비 강화
────────────────────────────────

분석 결과로 생성된 purchase_record는 오늘장사 장부로 보내기 전 검토 상태가 되어야 한다.

상태값:

accounting_status:
- pending: 반영 대기
- reviewed: 검토 완료
- synced: 장부 반영 완료
- excluded: 제외

이번 단계에서는 실제 오늘장사 장부 API 연동은 하지 않는다.
하지만 “장부 반영 준비 완료” 상태까지 만들어라.

구매내역 상세에서 표시:

> 이 구매내역은 오늘장사 장부에 반영할 준비가 완료되었습니다.

버튼:

- 장부 반영하기 UI
- 수정하기
- 제외하기

장부 반영하기 버튼은 이번 단계에서는 mock 처리해도 된다.

23. 완료 기준
────────────────────────────────

작업 완료 후 다음 기준을 만족해야 한다.

1. /app/analyze 화면에서 자료 업로드 또는 텍스트 입력을 시작할 수 있다.
2. 거래명세서/견적서/영수증/엑셀/카톡내용 등 자료 유형을 선택할 수 있다.
3. 업로드 후 분석 상태가 표시된다.
4. mock 분석 또는 실제 API 연결 준비 구조가 동작한다.
5. 분석 결과 검토 화면이 있다.
6. 추출된 품목을 수정/삭제/확정/제외할 수 있다.
7. 거래처명, 거래일, 공급가, 부가세, 총액을 표시할 수 있다.
8. 확인 필요한 항목이 별도로 표시된다.
9. 분석 결과를 견적요청으로 변환할 수 있다.
10. 분석 결과를 구매내역으로 저장할 수 있다.
11. 기존 단가/거래처 정보 공개 범위를 선택할 수 있다.
12. 분석 이력 화면이 있다.
13. 관리자 분석 모니터링 화면이 있다.
14. 구매내역이 오늘장사 장부 반영 준비 상태로 저장될 수 있다.
15. 기존 1~5단계 기능이 망가지지 않는다.
16. 모바일/PC 반응형이 깨지지 않는다.
17. 빌드 오류가 없어야 한다.

────────────────────────────────
26. 최종 목표 기억
────────────────────────────────

싸와!는 단순 견적앱이 아니다.

싸와!의 핵심은 다음이다.

> 사장님은 거래명세서나 견적서 사진만 올린다.
> 앱은 품목과 금액을 자동으로 정리한다.
> 사용자는 확인만 하고 견적요청을 만들 수 있다.
> 또는 구매내역으로 저장해 오늘장사 장부에 반영할 수 있다.
> 이후 매입비, 순이익, 세금자료 관리까지 연결된다.

이번 작업은 싸와!를 “직접 입력해야 하는 앱”에서 “자료만 올리면 자동 정리되는 앱”으로 바꾸는 핵심 단계다.

무리하게 완벽한 OCR 정확도를 목표로 하지 말고, 실제 서비스 확장이 가능한 UX와 데이터 구조를 안정적으로 구현해라.
