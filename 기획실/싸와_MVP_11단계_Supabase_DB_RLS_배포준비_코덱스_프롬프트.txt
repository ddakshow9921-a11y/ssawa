싸와! MVP 11단계 코덱스 작업지시서
주제: Supabase 실제 DB/RLS/Auth/Storage 점검 + 배포 준비 + 베타 테스트 환경 구축

너는 최고의 시니어 풀스택 개발자이자 보안 감각이 뛰어난 백엔드/인프라 개발자다.

우리는 B2B 자재 견적구매 플랫폼 “싸와!” 앱을 개발 중이다.

싸와!는 음식점, 카페, 매장, 공사현장, 사업자가 필요한 식자재, 포장재, 소모품, 설비자재, 건축자재 등을 올리면 지역 공급업체들이 견적을 제출하고, 구매자는 여러 견적을 비교해 업체를 선택할 수 있는 앱이다.

이번 11단계는 “실제 베타 테스트 환경 구축” 단계다.

지금까지는 기능 설계, mock 데이터, UI/UX, 운영 구조를 중심으로 개발했다.
이번 단계에서는 실제 사용자 베타 테스트를 위해 Supabase 데이터베이스, 인증, 권한, Storage, 환경변수, 배포 설정을 점검하고 정리한다.

중요:
이번 단계에서는 무리하게 모든 기능을 완벽한 운영 수준으로 바꾸는 것이 목적이 아니다.
목표는 “베타 테스트가 가능한 안전한 실제 환경”을 만드는 것이다.

3. 환경 구분
────────────────────────────────

앱은 다음 환경을 구분할 수 있어야 한다.

1. local
2. beta
3. production

환경변수 예시:

NEXT_PUBLIC_APP_ENV=local | beta | production
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
NEXT_PUBLIC_APP_URL=
NEXT_PUBLIC_ENABLE_DEMO_DATA=true | false
NEXT_PUBLIC_ENABLE_MOCK_AI=true | false
NEXT_PUBLIC_ENABLE_MOCK_PAYMENT=true | false
NEXT_PUBLIC_ENABLE_BETA_BADGE=true | false

주의:
SUPABASE_SERVICE_ROLE_KEY는 절대 클라이언트에 노출되면 안 된다.
브라우저 번들에 포함되지 않게 server-only 영역에서만 사용해라.

앱 상단 또는 관리자 화면에 beta 환경일 때 “베타 테스트” 배지를 표시해도 된다.

4. mock 데이터와 실제 데이터 분리
────────────────────────────────

지금까지 만든 mock/demo 데이터는 유지하되, 실제 DB 모드와 분리해야 한다.

권장 구조:

- lib/mock/*
- lib/demo/*
- lib/db/*
- lib/supabase/*

기능별로 다음처럼 선택 가능해야 한다.

1. mock mode:
   - 개발 중 빠른 UI 확인용

2. demo mode:
   - 베타 시연용 샘플 데이터

   - Supabase 실제 데이터 사용

NEXT_PUBLIC_USE_LIVE_DATA

공통 필드 권장:

- is_demo boolean default false
- created_by_seed boolean default false

────────────────────────────────
7. profiles 테이블
────────────────────────────────

profiles 구조:

- id uuid primary key references auth.users(id) on delete cascade
- email text
- name text
- phone text
- role text check role in ('buyer', 'supplier', 'admin')
- business_name text
- business_number text
- region text
- onboarding_completed boolean default false
- onboarding_completed_at timestamptz
- operational_status text default 'normal'
- is_demo boolean default false
- created_at timestamptz default now()
- updated_at timestamptz default now()

Auth 가입 시 profile 자동 생성 trigger 또는 앱 레벨 upsert를 구현한다.

권장:
handle_new_user() trigger를 만들 수 있다면 사용한다.
어렵다면 앱 로그인 직후 ensureProfile() 함수를 만들어 upsert한다.

supplier:
- 자신의 profile 조회/수정
- 자신의 supplier_profile 조회/수정

- 매칭되는 quote_requests 조회

admin:

- 수수료/정산 관리
- 운영/QA/피드백 관리

주의:
RLS에서 admin 판별 시 profiles.role = 'admin'을 사용하되, 무한 재귀가 발생하지 않도록 security definer function을 활용할 수 있다.

예시 함수:
is_admin()
get_my_role()

────────────────────────────────
9. RLS 기본 원칙
────────────────────────────────

모든 주요 테이블에 RLS를 활성화한다.

기본 원칙:

1. 본인 데이터만 조회 가능
2. 구매자는 자신의 quote_requests만 수정 가능
3. 공급업체는 승인된 상태에서만 quotes 생성 가능
4. 공급업체는 자신의 quotes만 수정 가능
5. deals는 buyer_id 또는 supplier_id가 본인인 경우 조회 가능
6. messages는 스레드 참여자만 조회/작성 가능
7. notifications는 본인 알림만 조회/수정 가능
8. purchase_records는 buyer 본인만 조회 가능
9. admin은 전체 접근 가능
10. public landing/정책 페이지는 DB 접근 없이 처리

RLS 정책 이름은 명확히 작성한다.

예시:
- profiles_select_own_or_admin
- quote_requests_select_owner_supplier_match_or_admin
- quotes_insert_approved_supplier
- deals_select_participants_or_admin
- messages_select_thread_participants_or_admin

────────────────────────────────
10. 공급업체 승인 상태 기반 접근 제어
────────────────────────────────

supplier_profiles.approval_status:

- pending
- approved
- needs_revision
- rejected
- suspended

공급업체 견적 제출 조건:

1. 로그인 사용자 role = supplier
2. supplier_profile.user_id = auth.uid()
3. supplier_profile.approval_status = approved
4. user operational_status not in suspended/banned
5. 요금제/견적 참여 한도 통과

UI에서도 제한:
- pending: 승인 대기 화면
- needs_revision: 보완 요청 화면
- rejected: 반려 화면
- suspended: 이용 제한 화면

서버/RLS에서도 최소한 approved supplier만 quotes insert 가능하도록 제한한다.

────────────────────────────────
11. 견적요청 조회 정책
────────────────────────────────

quote_requests 조회는 다음 조건을 고려한다.

buyer:
- 본인이 생성한 요청만 조회

supplier:
- 승인된 공급업체는 open/quoted 상태의 요청 중 자신의 카테고리/지역과 매칭되는 요청 조회 가능
- MVP에서는 RLS에서 복잡한 지역/카테고리 매칭을 완벽히 구현하기 어려우면 앱 서버/쿼리 단계에서 필터링하고, RLS는 approved supplier가 open 요청을 볼 수 있게 제한적으로 허용할 수 있다.
- 민감 정보는 거래 확정 전에는 제한한다.

admin:
- 전체 조회

중요:
거래 확정 전 구매자의 상세 주소/연락처는 공급업체에게 과도하게 노출하지 않는다.
필요하면 public_view용 필드를 분리하거나 UI에서 마스킹한다.

────────────────────────────────
12. 파일 Storage 설계
────────────────────────────────

Supabase Storage bucket을 설계한다.

권장 bucket:

1. quote-attachments
   - 견적요청 첨부파일
   - 거래명세서/견적서/품목사진

2. deal-attachments
   - 거래 증빙자료
   - 영수증/납품사진/세금계산서 파일

3. supplier-documents
   - 사업자등록증/통장사본/인증자료
   - 민감 자료이므로 private bucket 권장

4. analysis-files
   - OCR/AI 분석 대상 파일
   - 거래명세서/영수증/PDF/엑셀

5. feedback-attachments
   - 피드백/오류 신고 첨부파일

Storage 정책:

- 기본 private bucket 권장
- 파일 조회는 소유자 또는 관련 거래 참여자 또는 admin만 가능
- supplier-documents는 supplier 본인과 admin만 가능
- signed URL 방식 사용 권장
- public bucket은 로고/공개 이미지에만 사용

파일 경로 규칙:

quote-attachments/{userId}/{quoteRequestId}/{timestamp}_{filename}
deal-attachments/{userId}/{dealId}/{timestamp}_{filename}
supplier-documents/{supplierId}/{documentType}/{timestamp}_{filename}
analysis-files/{userId}/{analysisJobId}/{timestamp}_{filename}
feedback-attachments/{userId}/{feedbackId}/{timestamp}_{filename}

주의:
파일명에 특수문자/한글이 포함되면 스토리지 또는 배포 환경에서 문제가 생길 수 있다.
업로드 시 안전한 파일명으로 변환한다.

권장:
sanitizeFileName()
generateStoragePath()

19. 관리자 초기 세팅
────────────────────────────────

관리자 세팅 방식 중 하나를 구현하거나 문서화한다.

방법 A:
- Supabase Auth에서 관리자 이메일 가입
- SQL Editor에서 해당 user id의 profiles.role을 admin으로 변경

방법 B:
- ADMIN_EMAILS 환경변수에 있는 이메일은 ensureProfile 시 admin role 부여

방법 C:
- 관리자 seed SQL 제공

보안 주의:
운영환경에서는 아무나 admin role을 받을 수 없게 한다.
ADMIN_EMAILS 기반 자동 부여는 beta/local에서만 사용하는 것이 안전하다.

────────────────────────────────
31. 최종 목표 기억
────────────────────────────────

싸와!는 단순 견적앱이 아니다.

싸와!의 핵심은 다음이다.

> 구매자가 필요한 자재를 쉽게 올린다.
> 공급업체가 견적을 제출한다.
> 구매자가 비교 후 업체를 선택한다.
> 거래가 진행되고 구매내역이 정리된다.
> 오늘장사 장부와 연결되어 매입비와 순이익 관리까지 이어진다.
> 알림, 문의, 신고, 후기, 신뢰도, 정산 구조로 안전하게 운영된다.

이번 작업은 싸와!를 “로컬/mock 중심 앱”에서 “실제 베타 테스트 가능한 데이터 기반 앱”으로 전환하는 단계다.

무리하게 모든 기능을 완벽한 production 수준으로 만들지 말고, Supabase 실제 환경에서 안전하게 테스트할 수 있는 기반을 만드는 데 집중해라.
