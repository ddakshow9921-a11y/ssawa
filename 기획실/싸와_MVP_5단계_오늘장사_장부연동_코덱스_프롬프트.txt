싸와! MVP 5단계 Codex 작업지시서
오늘장사 장부 연동 + 구매내역 자동정리 + 매입비 리포트

너는 최고의 시니어 풀스택 개발자이자 UX 기획자다.

우리는 B2B 자재 견적구매 플랫폼 “싸와!” 앱을 개발 중이다.

싸와!는 음식점, 카페, 매장, 공사현장, 사업자가 필요한 식자재, 포장재, 소모품, 설비자재, 건축자재 등을 올리면 지역 공급업체들이 견적을 제출하고, 구매자는 여러 견적을 비교해 업체를 선택할 수 있는 앱이다.

지금까지의 핵심 흐름은 다음과 같다.

1단계:
- 싸와! 홈 화면
- 구매자 견적요청 작성
- 견적요청 목록/상세
- 공급업체 견적 제출
- 구매자 견적 비교
- 관리자 기본 관리

2단계:
- 견적요청 UX 고도화
- 사진/거래명세서/파일 업로드 UI
- 문장 입력 → 품목 리스트 변환 mock 기능
- AI 품목 정리 mock 화면
- 견적요청 템플릿
- 지난 요청 다시 사용하기
- 요청 완성도 점수
- 공급업체 매칭 예상 안내

3단계:
- 공급업체 입점 신청
- 사업자 정보 등록
- 취급 카테고리 설정
- 납품 가능 지역 설정
- 거래 조건 설정
- 인증자료 업로드 UI
- 관리자 공급업체 승인/반려
- 지역/카테고리 기반 요청 필터링
- 공급업체 프로필
- 공급업체 신뢰 정보 표시

4단계:
- 구매자 견적 선택
- 거래 생성
- 거래 상세 화면
- 구매자 거래내역
- 공급업체 거래관리
- 거래 상태 관리
- 거래 완료 처리
- purchase_records 구조 준비
- 예상 절감액 mock 계산
- 관리자 거래 모니터링

이번 작업은 “MVP 5단계”다.

이번 단계의 핵심 목표는 다음이다.

거래가 완료된 구매내역을 오늘장사 장부와 연결할 수 있는 구조로 만들고, 구매자가 식자재비, 포장재비, 소모품비, 설비자재비 등 매입비를 자동으로 확인할 수 있게 만든다.

싸와!는 단순 견적앱이 아니라, “자재 구매 → 거래 완료 → 매입 자동정리 → 순이익 분석 → 세금자료 정리”로 이어지는 사업자 운영 플랫폼이 되어야 한다.

이번 단계에서는 실제 홈택스 연동, 실제 세금 신고, 실제 세금계산서 발행은 구현하지 않는다.
하지만 향후 연동 가능하도록 데이터 구조, 화면, 상태값, UI 흐름을 정확하게 설계한다.

────────────────────────────────────
1. 작업 전 필수 확인
────────────────────────────────────

작업 전에 반드시 현재 프로젝트 구조를 먼저 분석해라.

확인할 것:

1. 현재 프레임워크와 라우팅 구조
2. 기존 싸와! 1~4단계 구현 파일
3. 거래 deals/orders 데이터 구조
4. 구매내역 purchase_records 데이터 구조
5. 견적요청 quote_requests 데이터 구조
6. 공급업체 supplier_profiles 데이터 구조
7. 사용자 역할 buyer / supplier / admin 처리 방식
8. mock 데이터 또는 Supabase 사용 여부
9. 오늘장사 관련 기존 코드/페이지/컴포넌트 존재 여부
10. 스타일링 방식
11. 모바일 반응형 상태
12. 빌드 오류 여부

기존 기능을 삭제하거나 망가뜨리지 마라.

만약 오늘장사 장부 관련 기존 기능이 이미 있다면 해당 구조를 분석한 뒤 최대한 재사용해라.
만약 오늘장사 장부 기능이 아직 없다면, 이번 단계에서는 “오늘장사 연동 준비용 장부/매입관리 화면”을 싸와! 내부에 구현해라.

────────────────────────────────────
2. 이번 단계 핵심 기능
────────────────────────────────────

이번 MVP 5단계에서 구현할 핵심 기능은 다음이다.

1. 거래 완료 후 구매내역 자동 생성

3. 구매내역 상세 화면
4. 구매내역 카테고리 자동 분류
5. 매입비 요약 대시보드
6. 월별 매입 리포트
7. 카테고리별 지출 분석
8. 공급업체별 구매액 분석
9. 예상 절감액 리포트
10. 세금계산서/영수증/거래명세서 상태 관리
11. 오늘장사 장부 반영 상태 관리
12. 구매내역 수동 수정 UI
13. 구매내역 제외/보류 처리
14. 관리자 구매내역 모니터링
15. 추후 세금/순이익 분석으로 확장 가능한 구조

이번 단계에서는 “실제 회계/세무 자동처리”가 아니라 “장부 연동이 가능한 구매내역 정리 구조”를 만드는 것이 핵심이다.

────────────────────────────────────
3. 전체 사용자 흐름
────────────────────────────────────

이번 단계 이후 사용자는 다음 흐름을 경험해야 한다.

1. 구매자가 싸와!에서 견적요청을 등록한다.
2. 공급업체가 견적을 제출한다.
3. 구매자가 견적을 선택한다.
4. 거래가 생성된다.
5. 거래가 완료된다.
6. 완료된 거래가 구매내역으로 자동 저장된다.
7. 구매내역은 카테고리별 매입으로 분류된다.
8. 구매자는 이번 달 식자재비/포장재비/소모품비 등을 확인한다.
9. 세금계산서, 영수증, 거래명세서 상태를 확인한다.
10. 오늘장사 장부 반영 대기/완료 상태를 확인한다.
11. 월별 매입비와 예상 절감액 리포트를 확인한다.

핵심 메시지:

“싸와!에서 구매하면 오늘장사 장부에 자동으로 정리됩니다.”

────────────────────────────────────
6. 거래 완료 → 구매내역 자동 생성
────────────────────────────────────

4단계에서 거래가 completed 상태가 되면 purchase_records가 생성될 수 있는 구조를 만들었다.
이번 단계에서는 이 흐름을 더 명확하게 구현한다.

필수 함수 예시:

- createPurchaseRecordFromDeal(deal)
- createPurchaseRecordItemsFromDealItems(dealItems)
- calculateSupplyAndVat(finalAmount)
- inferAccountingCategory(categoryName, items)
- calculateEstimatedSavings(previousAmount, selectedAmount)
- createAccountingEntryFromPurchaseRecord(purchaseRecord)
- markPurchaseAsAccountingPending(purchaseRecordId)

동작 흐름:

1. deal 상태가 completed로 변경된다.
2. 해당 deal에 연결된 quote_request, selected_quote, deal_items를 기반으로 purchase_record를 생성한다.
3. deal_items를 purchase_record_items로 복사한다.
4. 카테고리 기준으로 accounting_category를 자동 지정한다.
5. 공급가/부가세를 계산한다.
6. 세금계산서 필요 여부에 따라 tax_invoice_status를 설정한다.
7. accounting_status를 pending으로 설정한다.
8. 오늘장사 장부 반영 대기 목록에 표시한다.

중복 생성 방지:

이미 같은 deal_id로 purchase_record가 있으면 새로 만들지 말고 기존 데이터를 사용한다.

────────────────────────────────────
7. 구매내역 카테고리 자동 분류
────────────────────────────────────

싸와! 카테고리를 오늘장사 장부용 카테고리로 자동 분류한다.

기본 매핑:

- 식자재 → 식자재비
- 포장재 → 포장재비
- 소모품 → 소모품비
- 주방용품 → 주방용품비
- 설비/닥트/환기자재 → 설비자재비
- 건축자재 → 건축자재비
- 공구/산업자재 → 공구/산업자재비
- 기타 → 기타매입

함수 예시:

getAccountingCategory(categoryName)

사용자가 구매내역 상세에서 카테고리를 수동 변경할 수 있어야 한다.

수동 변경 시:

- purchase_records.accounting_category 업데이트
- purchase_record_items.accounting_category 업데이트 가능
- 리포트에도 반영

────────────────────────────────────
10. 장부 반영 상태 관리
────────────────────────────────────

purchase_records.accounting_status 기준으로 상태를 관리한다.

상태값:

- pending: 반영 대기
- synced: 장부 반영 완료
- excluded: 장부 제외
- hold: 보류
- failed: 반영 실패

사용자에게 표시하는 한국어:

- 반영 대기
- 반영 완료
- 제외됨
- 보류 중
- 반영 실패

상태 변경 흐름:

1. 거래 완료 후 purchase_record 생성 → pending
2. 사용자가 “장부 반영하기” 클릭 → accounting_entry 생성 → synced
3. 사용자가 “제외하기” 클릭 → excluded
4. 사용자가 “보류하기” 클릭 → hold
5. 오류 발생 mock → failed

상태 변경 시 안내 문구:

- “오늘장사 장부에 반영되었습니다.”
- “이 구매내역은 장부에서 제외되었습니다.”
- “이 구매내역은 보류 처리되었습니다.”

24. 완료 기준
────────────────────────────────────

작업 완료 후 다음 기준을 만족해야 한다.

1. 거래 완료 시 purchase_record가 생성된다.
2. purchase_record_items가 생성된다.
3. 구매내역 목록 화면이 있다.
4. 구매내역 상세 화면이 있다.
5. 구매내역이 카테고리별 장부 분류로 자동 매핑된다.
6. 사용자가 장부 분류를 수정할 수 있다.
7. 세금계산서/영수증/거래명세서 상태가 표시된다.
8. 증빙자료 업로드 UI가 있다.
9. 장부 반영 대기/완료/제외/보류 상태가 관리된다.
10. “오늘장사 장부에 반영” UI가 있다.
11. 매입비 요약 대시보드가 있다.
12. 월별 매입 리포트가 있다.
13. 카테고리별 구매액 분석이 있다.
14. 공급업체별 구매액 분석이 있다.
15. 예상 절감액 리포트가 있다.
16. 구매내역 수동 추가 UI가 있다.
17. 관리자 구매내역 모니터링 화면이 있다.
18. 기존 1~4단계 기능이 망가지지 않는다.
19. 모바일/PC 반응형이 깨지지 않는다.
20. 빌드 오류가 없어야 한다.

25. 작업 완료 후 보고 형식
────────────────────────────────────

작업이 끝나면 아래 형식으로 보고해라.

1. 현재 프로젝트 구조 분석 결과
2. 수정/추가한 파일 목록
3. 구현한 구매내역/장부 연동 기능 목록
4. 추가한 화면 목록
5. 추가/변경한 타입 또는 데이터 구조
6. 거래 완료 → 구매내역 생성 흐름 설명
7. 장부 반영 상태 흐름 설명
8. 매입비 리포트 계산 방식 설명
9. 테스트 방법
10. 남은 문제점
11. 다음 단계 추천 작업

26. 다음 단계 예고
────────────────────────────────────

이번 5단계가 끝나면 다음 6단계에서는 아래 기능을 구현할 예정이다.

- 순이익 분석 대시보드
- 매출 대비 매입비 분석
- 카테고리별 비용 비율
- 예상 부가세 참고 계산
- 예상 종합소득세/법인세 참고 구조
- AI 절감 리포트
- 월별 사업 성과 리포트
- “이번 달 장사가 남는지” 한눈에 보여주는 화면

이번 단계에서는 다음 단계 확장을 막지 않도록 purchase_records, accounting_entries, monthly_purchase_summaries 구조를 신중하게 설계해라.

────────────────────────────────────
27. 최종 목표 기억
────────────────────────────────────

싸와!와 오늘장사의 최종 목표는 다음이다.

“사장님이 필요한 자재를 싸와!에서 견적받고 구매한다.
구매내역은 자동으로 오늘장사 장부에 정리된다.
사장님은 이번 달 식자재비, 포장재비, 소모품비, 총 매입비를 바로 확인한다.
이후 매출과 연결해 순이익, 예상 세금, 절감 포인트까지 확인한다.”

이번 작업은 이 최종 목표에서 “구매내역 자동정리와 장부 반영 준비”를 완성하는 핵심 단계다.

무리하게 세금 신고나 실제 홈택스 연동까지 구현하지 말고, 거래 완료 후 구매내역이 정확히 정리되고, 오늘장사 장부로 연결될 수 있는 안정적인 기반을 만들어라.
