싸와! MVP 7단계 코덱스 작업지시서
주제: 알림센터 + 견적/거래 상태 알림 + 구매자·공급업체 문의 스레드

너는 최고의 시니어 풀스택 개발자이자 UX 기획자다.

우리는 B2B 자재 견적구매 플랫폼 “싸와!” 앱을 개발 중이다.

싸와!는 음식점, 카페, 매장, 공사현장, 사업자가 필요한 식자재, 포장재, 소모품, 설비자재, 건축자재 등을 올리면 지역 공급업체들이 견적을 제출하고, 구매자는 여러 견적을 비교해 업체를 선택할 수 있는 앱이다.

이전 단계까지 다음 기능을 구현했다.

- 관리자 분석 모니터링

이번 작업은 MVP 7단계다.

이번 단계의 핵심 목표는 “앱 안에서 중요한 변화를 놓치지 않게 알려주는 구조”와 “거래 전 필요한 문의를 안전하게 주고받는 구조”를 만드는 것이다.

싸와!는 견적요청, 견적 제출, 견적 선택, 거래 수락, 납품 준비, 거래 완료, 분석 완료 등 상태 변화가 많다.
따라서 구매자와 공급업체가 앱을 계속 새로고침하지 않아도 중요한 내용을 확인할 수 있도록 알림센터와 상태 알림을 구축해야 한다.

또한 실제 실시간 채팅까지 무리하게 구현하기보다는, MVP에서는 “문의 스레드” 형태로 구매자와 공급업체가 견적/거래 단위로 메시지를 남길 수 있는 구조를 만든다.

이번 단계에서는 실제 카카오 알림톡, 문자, 이메일 발송, Web Push 연동은 구현하지 않는다.
단, 추후 외부 알림 채널을 붙일 수 있도록 notification_events 구조와 사용자 알림 설정을 설계한다.

5. 알림 타입 정의
────────────────────────────────

알림 type은 다음과 같이 정의한다.

구매자 알림:

- quote_received: 새 견적 도착
- quote_updated: 견적 수정됨
- quote_expiring: 견적 유효기간 임박
- supplier_message_received: 공급업체 문의 도착
- deal_created: 거래 생성됨
- deal_confirmed: 공급업체가 거래 수락
- deal_preparing: 납품 준비 시작
- deal_delivering: 배송/납품 시작
- deal_delivered: 납품 완료 처리됨
- deal_completed: 거래 완료됨
- deal_cancelled: 거래 취소됨
- deal_disputed: 문제 신고 접수
- analysis_completed: 자료 분석 완료
- analysis_needs_review: 분석 결과 검토 필요
- purchase_record_created: 구매내역 생성됨
- accounting_sync_ready: 오늘장사 장부 반영 준비 완료

공급업체 알림:

- new_matched_request: 새 매칭 견적요청
- request_updated: 견적요청 내용 변경
- quote_selected: 내 견적이 선택됨
- quote_rejected: 내 견적이 미선택됨
- buyer_message_received: 구매자 문의 도착
- deal_waiting_confirmation: 거래 확인 대기
- deal_cancelled_by_buyer: 구매자 거래 취소
- deal_completed_by_buyer: 구매자 거래 완료 확인
- document_review_result: 입점/인증자료 검토 결과

관리자 알림:

- supplier_apply_submitted: 공급업체 입점 신청
- supplier_document_uploaded: 인증자료 업로드
- high_value_deal_created: 고액 거래 생성
- deal_dispute_created: 문제 신고 발생
- analysis_failed: 분석 실패 다수 발생
- message_reported: 문의/메시지 신고
- system_error: 시스템 오류 mock

시스템/마케팅 알림은 이번 단계에서 선택적으로 구현한다.

────────────────────────────────
6. 알림 생성 규칙
────────────────────────────────

다음 이벤트가 발생할 때 알림을 생성해야 한다.

1. 공급업체가 견적 제출
   - 구매자에게 quote_received 알림 생성

2. 구매자가 견적 선택
   - 선택된 공급업체에게 quote_selected 알림 생성
   - 미선택 공급업체에게 quote_rejected 알림 생성
   - 구매자에게 deal_created 알림 생성

3. 거래가 생성됨
   - 공급업체에게 deal_waiting_confirmation 알림 생성

4. 공급업체가 거래 수락
   - 구매자에게 deal_confirmed 알림 생성

5. 공급업체가 납품 준비 시작
   - 구매자에게 deal_preparing 알림 생성

6. 공급업체가 배송/납품 시작
   - 구매자에게 deal_delivering 알림 생성

7. 공급업체가 납품 완료 처리
   - 구매자에게 deal_delivered 알림 생성

8. 구매자가 거래 완료 처리
   - 공급업체에게 deal_completed_by_buyer 알림 생성
   - 구매자에게 purchase_record_created 알림 생성

9. 거래 취소
   - 상대방에게 deal_cancelled 알림 생성

10. 문제 신고
   - 상대방과 관리자에게 deal_disputed 알림 생성

11. 분석 완료
   - 사용자에게 analysis_completed 알림 생성

12. 분석 검토 필요
   - 사용자에게 analysis_needs_review 알림 생성

13. 구매내역이 오늘장사 장부 반영 준비 상태가 됨
   - 구매자에게 accounting_sync_ready 알림 생성

14. 공급업체 입점 신청
   - 관리자에게 supplier_apply_submitted 알림 생성

15. 인증자료 업로드
   - 관리자에게 supplier_document_uploaded 알림 생성

16. 문의 메시지 등록
   - 상대방에게 message_received 알림 생성

가능하면 helper 함수를 만든다.

권장 함수:

- createNotification(input)
- createNotificationEvent(input)
- markNotificationAsRead(id)
- markAllNotificationsAsRead(userId)
- getUnreadNotificationCount(userId)
- getNotificationsForUser(userId)
- generateNotificationFromEvent(event)
- createQuoteReceivedNotification(quote)
- createDealStatusNotification(deal, status)
- createAnalysisNotification(analysisJob)
- createMessageNotification(message)

- last_read_at
- unread_count
- updated_at

message_reports

- id
- thread_id
- message_id
- reported_by
- reason
- detail
- status: pending | reviewed | resolved | dismissed
- created_at
- updated_at

실제 채팅 서버나 WebSocket은 이번 단계에서 구현하지 않는다.
새로고침 또는 mock state 기반으로 동작해도 된다.

────────────────────────────────
11. 견적요청 문의 스레드
────────────────────────────────

/app/requests/[id]/messages 화면을 만든다.

목적:
구매자가 특정 견적요청에 대해 공급업체와 질문/답변을 주고받을 수 있게 한다.

주의:
구매자가 여러 공급업체와 동시에 문의할 수 있으므로, 공급업체별 스레드가 분리되어야 한다.

예시:

견적요청: 치킨집 포장재 견적 요청

스레드 1:
- 서울포장과 문의

스레드 2:
- 경기소모품센터와 문의

화면 구성:

- 견적요청 요약
- 공급업체별 문의 목록
- 메시지 영역
- 입력창
- 파일 첨부 UI
- 신고 버튼 UI

구매자 문구:

> 업체에 추가 조건을 문의해보세요.
> 개인정보나 외부 결제 요청은 주의해주세요.

공급업체 문구:

> 구매자의 요청 조건을 확인하고 필요한 내용을 문의하세요.

────────────────────────────────
12. 거래 문의 스레드
────────────────────────────────

/app/deals/[id]/messages 화면을 만든다.

거래가 생성된 후에는 구매자와 선택된 공급업체 간 문의 스레드를 제공한다.

사용 목적:

- 납품 시간 조율
- 배송 위치 확인
- 품목 변경 문의
- 세금계산서 요청
- 증빙자료 확인
- 문제 발생 전 협의

화면 구성:

- 거래 요약 카드
- 거래 상태
- 메시지 목록
- 입력창
- 첨부파일 UI
- 문제 신고 버튼
- 거래 상세로 돌아가기 버튼

거래 상태와 함께 메시지가 보여야 한다.

예시 시스템 메시지:

- 거래가 생성되었습니다.
- 공급업체가 거래를 수락했습니다.
- 납품 준비가 시작되었습니다.
- 공급업체가 납품 완료 처리했습니다.
- 구매자가 거래 완료를 확인했습니다.

시스템 메시지도 messages에 sender_role: system으로 저장하거나 mock 표시한다.

13. 메시지 작성/읽음 처리
────────────────────────────────

메시지 작성 기능:

입력값:

- 본문
- 첨부파일 UI
- 전송 버튼

전송 시:

1. messages에 메시지 생성
2. thread last_message_at 업데이트
3. 상대방에게 message notification 생성
4. 상대방 unread_count 증가
5. 화면에 즉시 표시

읽음 처리:

- 사용자가 스레드에 들어오면 해당 thread의 unread_count를 0으로 처리
- 관련 메시지 is_read 또는 read state 업데이트

MVP에서는 실제 DB 없이 mock state로 처리 가능하다.

16. 공급업체 신규 요청 알림

17. 견적 유효기간 임박 알림
────────────────────────────────

견적 유효기간이 임박한 경우 알림을 표시할 수 있는 구조를 만든다.

이번 단계에서는 실제 cron이나 스케줄러를 구현하지 않는다.
대신 mock 함수 또는 관리자/화면 진입 시 계산으로 표시한다.

- 구매자가 아직 선택하지 않은 견적

공급업체 알림은 선택사항이다.

함수 예시:

- getExpiringQuotes(quotes)
- createQuoteExpiringNotifications()

────────────────────────────────
19. 알림 우선순위
────────────────────────────────

priority를 UI에 반영한다.

low:
- 일반 안내

normal:
- 일반 견적/분석 알림

high:
- 거래 상태 변경, 검토 필요

urgent:
- 문제 신고, 거래 취소, 관리자 조치 필요

UI 표시:

- urgent는 빨간색 계열 경고 배지
- high는 오렌지 계열 강조
- normal은 기본
- low는 흐린 회색 또는 보조 스타일

너무 강한 색을 남발하지 말고, 싸와! 브랜드 톤을 유지한다.

────────────────────────────────
20. 알림 공개 범위/보안
────────────────────────────────

알림에는 민감한 정보가 과도하게 노출되지 않게 한다.

주의사항:

- 기존 거래처 단가는 알림 본문에 표시하지 않는다.
- 사업자등록번호 전체를 알림 본문에 표시하지 않는다.
- 거래 상세 금액은 사용자 본인의 알림에만 표시한다.
- 공급업체에게 구매자의 상세 주소는 거래 확정 전에는 표시하지 않는다.
- 관리자는 전체 정보를 볼 수 있지만, 화면에서 필요한 정보만 표시한다.

알림 본문 예시:

좋은 예:
> 새 견적이 도착했습니다. 견적 상세에서 금액을 확인하세요.

피해야 할 예:
> 서울포장이 기존 거래처보다 38,200원 싸게 견적을 냈습니다.

21. UI/디자인 방향
────────────────────────────────

싸와! 브랜드 느낌을 유지한다.

디자인 방향:

- 오렌지/코랄 메인 컬러
- 딥 네이비 보조 컬러

22. 더미 데이터
────────────────────────────────

개발 확인용 더미 데이터를 추가한다.

구매자 알림 샘플:

1. 새 견적 도착
- 제목: 새 견적이 도착했어요.
- 내용: 서울포장이 치킨집 포장재 견적 요청에 견적을 제출했습니다.
- type: quote_received
- priority: normal

2. 거래 수락
- 제목: 공급업체가 거래를 수락했어요.
- 내용: 서울포장이 거래를 수락했습니다. 납품 일정을 확인해주세요.
- type: deal_confirmed
- priority: high

3. 분석 검토 필요
- 제목: 분석 결과 확인이 필요해요.
- 내용: 거래명세서에서 수량이 불명확한 품목 2개가 있습니다.
- type: analysis_needs_review
- priority: high

공급업체 알림 샘플:

1. 새 요청 도착
- 제목: 새 견적요청이 도착했어요.
- 내용: 서울 동대문구 포장재 요청에 견적을 제출할 수 있습니다.
- type: new_matched_request
- priority: normal

2. 내 견적 선택
- 제목: 내 견적이 선택되었습니다.
- 내용: 강남치킨이 서울포장의 견적을 선택했습니다. 거래 확인이 필요합니다.
- type: quote_selected
- priority: high

관리자 알림 샘플:

1. 입점 신청
- 제목: 공급업체 입점 신청이 들어왔습니다.
- 내용: 서울포장이 입점 신청을 완료했습니다.
- type: supplier_apply_submitted
- priority: normal

2. 문제 신고
- 제목: 거래 문제 신고가 접수되었습니다.
- 내용: 고깃집 식자재 거래에서 문제가 신고되었습니다.
- type: deal_disputed
- priority: urgent

메시지 샘플:

견적요청 문의:
- 구매자: 소스컵은 투명 제품으로 가능한가요?
- 공급업체: 네, 투명 2온스 기준으로 견적드렸습니다.
- 구매자: 세금계산서도 가능한가요?
- 공급업체: 가능합니다.

거래 문의:
- 공급업체: 내일 오전 10시 전 납품 가능합니다.
- 구매자: 가게 뒷문으로 납품 부탁드립니다.
- 시스템: 공급업체가 납품 준비를 시작했습니다.

────────────────────────────────
27. 최종 목표 기억
────────────────────────────────

싸와!는 단순 견적앱이 아니다.

싸와!의 핵심은 다음이다.

> 사장님이 필요한 자재를 올린다.
> 공급업체들이 견적을 제출한다.
> 중요한 변화가 생기면 앱이 바로 알려준다.
> 거래 전 필요한 문의를 안전하게 주고받는다.
> 거래 상태와 구매내역이 관리된다.
> 이후 오늘장사 장부, 순이익 분석, 세금자료 정리와 연결된다.

이번 작업은 싸와!를 “사용자가 직접 계속 확인해야 하는 앱”에서 “중요한 상황을 알아서 알려주는 앱”으로 바꾸는 단계다.

무리하게 외부 알림 발송이나 실시간 채팅까지 구현하지 말고, 앱 안 알림센터와 문의 스레드 구조를 안정적으로 만들어라.
